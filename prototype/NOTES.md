# Прототип: билдер урока (throwaway)

**Вопрос:** какая раскладка степ-листа удобнее автору для сборки урока из шагов
(`text/video/material/task/work/assessment`)?

**Артефакт:** `lesson-builder.prototype.html` — 3 структурно разных варианта, переключатель `?variant=A|B|C`
(стрелки в плавающей панели снизу или ← → на клавиатуре).

Открыть (плагин смонтирован в WP):
```
http://localhost:8080/wp-content/plugins/fs-lms/prototype/lesson-builder.prototype.html
```
Файл автономный (HTML+CSS+JS inline), стейт в памяти, сбрасывается на reload. Это **не** продакшен-код.

## Варианты
- **A — Лента шагов.** Одна колонка, правка в карточке (Stepik-style). Линейный поток виден целиком.
- **B — Мастер-деталь.** Слева список шагов, справа редактор выбранного (Tutor-style). Меньше скролла.
- **C — Структура курса.** Всё дерево модуль→урок→шаги. Для архитектуры курса и баланса модулей.

Во всех: модалка «Добавить шаг» (выбор типа; для «задачи» — источник Мои/Банк + тип из реестра + библиотека/создать),
инлайн-разворот work/assessment в список задач (12 задач = 12 строк), drag-drop порядок, пример «Контрольная по 7».

## Что щупаем
1. На каком уровне держать фокус автора: один урок (A/B) или весь курс (C)?
2. Достаточно ли инлайн-разворота чипа работы/контрольной, или нужен переход внутрь на её собственный билдер?
3. Модалка добавления — хватает ли двух шагов (тип → детали) или избыточна?

## Вердикт (2026-06-18)

Микс по уровням:
- **Билдер урока → вариант B (мастер-деталь):** слева список шагов, справа редактор выбранного. Привычно.
  Дизайн дорабатывается позже. Живёт на экране редактирования `{key}_lessons`.
- **Страница курса → вариант C (структура курса):** дерево модуль→урок→шаги на экране редактирования
  `{key}_courses` (нативная админка, раздел «Обучение → Курсы»). Уроки разворачиваются, шаги — read-only
  превью; «редактировать урок» уводит на его B-билдер (drill-down).
- Вариант A — отклонён.

Работы/контрольные (`{key}_works`/`{key}_assessments`) — тот же B-билдер, что и урок (рельса шагов +
редактор), но `allowedTypes` = только `task`.

**Архитектура (2026-06-18): нативный drill-down, без SPA** («больше нативного wp, меньше кода»).
Каждая сущность — свой нативный WP-экран (метабокс-билдер вместо редактора): курс = C-дерево, урок/работа/
контрольная = B-мастер-деталь. Переходы между ними — штатная навигация WP (ссылка из дерева курса на экран
урока + «← к курсу»). Единый full-screen SPA-конструктор **отклонён** (дороже, уходит от нативного WP,
прячет переиспользование). «Рельса курса слева на экране урока» — опционально, позже (не в MVP, экономим код).
