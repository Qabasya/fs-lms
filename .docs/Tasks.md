### Задача 3 — «распаковка» ThreeInOneTemplate в экзаменах

**Текущее поведение — обнаружен реальный функциональный баг, не только визуальный:**
- Серверная проверка ответов УЖЕ разворачивает Triple-задание на 3 подпункта: `BatchCheckService.php:67-96` + `ThreeInOneTemplate::expandsForExam()` (`inc/MetaBoxes/Templates/ThreeInOneTemplate.php:76-82`) — ожидает `$studentAnswers` МАССИВОМ с ключами `'19'/'20'/'21'` (строки 71, 76).
- Но фронтенд рендерит Triple-задание как ОДИН блок `.fs-attempt-question` с ОДНИМ общим текстовым полем `answer_{taskId}` (`templates/frontend/assessment/partials/attempt-question.php:21-49`), а условия трёх подзаданий склеиваются в один HTML через `AssessmentPageController::buildCondition()` (строки 314-329, `<div class="fs-attempt-subcondition">` × 3).
- Итог: при сдаче Triple-задания в экзамене отправляется обычная строка, а не массив по ключам `19/20/21` → `is_array($answer) ? $answer : []` даёt пустой массив → **все 3 подответа всегда засчитываются как пустые/неверные**, независимо от того, что реально ввёл ученик.
- Нумерация (`attempt-form-nav.php:19-30`, `attempt-question.php:24`) считает по taskId, а не по развёрнутым подпунктам — один Triple-task занимает одну позицию в списке/навигаторе, а не три последовательных номера.

**Этапы:**
1. В `AssessmentPageController::buildTaskViews()`/`buildCondition()` — для Triple в контексте `kind->expandsComposites()` не склеивать условия в один блок, а разворачивать на этапе построения `$taskViews` в 3 отдельных виртуальных пункта (свои условие + номер).
2. В `attempt-question.php` и `attempt-form-nav.php` — рендерить составную задачу как 3 независимых `.fs-attempt-question` блока с полями ответа, соответствующими ключам, которые ждёт `BatchCheckService::check()` (`answer[{taskId}][19]` и т.п., либо привести `BatchCheckService` к реальному формату полей — выбрать один контракт и синхронизировать обе стороны).
3. Пересчитать сквозную нумерацию: Triple на «слоте 1» → 3 последовательных номера, следующее обычное задание — со следующим номером (учесть в `fs-ege-nav` и `attempt-question.php:24`, `$i+1`).
4. На этапе сборки ответа на клиенте (JS, отправляющий `FormData` формы попытки) — собрать под-ответы Triple в структуру, которую реально ожидает `BatchCheckService`.
5. Баллы: по 1 баллу за каждое из 19/20/21 по умолчанию — это уже поддерживается (`BatchCheckService.php:75`, `taskPoints["{$taskId}:{$sub['key']}"]`, см. также задачу 7 про дефолт баллов).
6. Регресс-тест: Triple под номером N + обычное задание под N+1 → после разворота 1..3(19-21), 4(следующее); итоговый балл = сумма верно решённых подзаданий.
