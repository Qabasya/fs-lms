# FS LMS — Backlog замечаний по плееру и /profile/ (план реализации)

Источник — 8 замечаний по итогам Эпика 14 (плеер курса) и кабинета `/profile/`.
Первопричины подтверждены анализом кода (2026-07-04). Ниже — дословные
замечания, зафиксированные решения по развилкам и план по каждому пункту с
точками правки.

## Зафиксированные решения (2026-07-04)

- **D-1 (задача 1) — календарь индивидуальных занятий отдельный, не привязан к
  группе.** На одном календаре показываются ВСЕ индивидуальные занятия всех
  групп/учеников. Пример: занятие Вани (Группа 1) в среду 12:00 и Маши (Группа 2)
  в среду 14:00 — оба видны на одном календаре.
- **D-2 (задача 5) — dry-run покрывает всё, включая контрольные и экзамены.**
  Преподаватель/методист/админ должен иметь возможность прорешать в предпросмотре
  задания, работы И контрольные/экзамены. Результат нигде не сохраняется.
- **D-3 (задача 7) — сайдбар плеера при входе учителя не меняется вовсе.**
  Показывается полная штатная навигация кабинета (как в `/profile/`), без блока
  «Сейчас проходите» и без потери кнопки меню/выхода.
- **D-4 (задача 3b) — поведение блокировки урока по времени.** Пока до начала
  больше часа — статичное «Урок ещё не доступен». За час и ближе — «Занятие скоро
  начнётся» + таймер обратного отсчёта до начала; по достижении времени урок
  разблокируется.

### B1 — profGradePop: ширина ≥ 500, клэмп относительно экрана — **XS**

**Причина:** ширина захардкожена в двух местах, и клэмп по горизонтали использует
устаревшую константу 188. Широкий контент (форма инд. занятия `openIndiForm`) не
влезает в 188px → уезжает за правый край и режется.
- `src/scss/profile/components/_overlays.scss:50` — `width: 188px;` → заменить на
  `min-width: 500px; max-width: calc(100vw - 20px);` (оставить `overflow-y:auto`,
  `max-height` из :51).
- `src/js/profile/utils.js:202` — `const pw = 188;` → `const pw = pop.offsetWidth;`
  (`.open` уже добавлен на :200, ширина измерима). Клэмп на :204
  (`Math.min(left, window.innerWidth - pw - 10)`) тогда сам держит бокс на экране;
  центрирование :203 и `Math.max(10, …)` уже делают left относительным.

Делать вместе/перед B2 — это та же форма.

## Зафиксированные решения (2026-07-05)

- **D-A — доставка через «открытую группу»** (`groups.access_mode='open'`), а не
  скрытую техгруппу и не отдельный бес-групповой контур. Причина: сдачи работ,
  журнал, кокпит, будущие дедлайны/ручные проверки уже ключуются по группе /
  `group_lesson_id` — всё достаётся бесплатно. Отвергнуты: скрытая техгруппа
  (фантомная сущность), отдельный контур `course_access`+`course_progress`
  (дублирование сдач/журнала/записи при требованиях «работы + наблюдение»).
- **D-B — никаких дата-хаков** («все уроки открываются 1999-01-01» не нужно):
  `LessonGateResolver::dateReached()` уже трактует `scheduled_at=NULL` как
  «открыто». Программа открытой группы публикуется целиком с `visibility='open'`
  и без дат.
- **D-C — работы разрешены, но только полностью автопроверяемые** (у каждой
  задачи есть правильный ответ). Работы с ручной проверкой — запрет валидацией.
  Контрольные (`assessment`) на первом этапе — по тому же правилу.
- **D-D — наблюдение за прохождением** — штатные кокпит/журнал группы:
  «дефолтный преподаватель» = `groups.teacher_id` (свой или технический аккаунт),
  отдельный дашборд не нужен.
- **D-E — запись ученика = вторая параллельная лёгкая запись `student_records`.**
  Записей у ученика может быть несколько (`findActiveByStudent()` — массив);
  договор/приказ/снапшоты в DDL — `DEFAULT NULL`. Основная запись (ЕГЭ) не
  трогается, «возврат из архива» не нужен. Родитель копируется из существующей
  записи ученика.
- **D-F — инвариант «создание записи ⇒ событие зачисления» сохраняется.**
  Лёгкий экшен обязан диспатчить `LogEvent::StudentEnrolled` +
  `EnrollmentStatusEvent` (→ журнал аудита) + `do_action('fs_lms_student_enrolled')`

  существующего ученика — ТОЛЬКО в группы `access_mode='open'`. Путь в
  расписаночные группы один — полный флоу зачисления; перевод (КЕГЭ-1 → КЕГЭ-2 =
  отчисление «Перевод» + новое зачисление) не меняется.
