## Эпик 18: предметы без собственного банка заданий/статей (DONE)

> **Статус:** реализовано, T18.1–T18.7 закрыты. Полный прогон тестов зелёный (797/797), live-проверка через `wp eval` (короткий тестовый ключ `tbank`) подтвердила корректную ветвление регистрации CPT/таксономии.
> **Дата:** 2026-07-07.
> **Триггер:** пункт 7 выше — открытые/бесплатные курсы (Git, работа с ЛМС, робототехника и т.п.), у которых нет и не будет собственного банка заданий/статей.

### Цель

Разрешить создавать предмет с `tasks_count = 0`, у которого:
- не регистрируются CPT `{key}_tasks` / `{key}_articles` и фиксированная таксономия `{key}_task_number`;
- нет подстраницы в верхнем меню wp-admin «Предметы» (`SubjectsMenuBuilder`) — она сейчас всегда рендерит таблицы заданий/статей этого предмета, для безбанкового предмета она бессмысленна;
- нет вкладок «Задания предмета» / «Статьи предмета» под «Обучение» (`LearningMenuController`);
- группы, зачисление, курсы/уроки/работы по этому предмету работают как обычно (CPT `lessons`/`works`/`courses`/`assessments` предмета не трогаем — они остаются, т.к. лекционные/видео-курсы всё ещё нужны).

### Зафиксированные решения

- **D18.1 — Флаг только при создании.** «Безбанковость» предмета выставляется один раз при создании (поле `tasks_count = 0`) и не редактируется после. Превращение существующего предмета в безбанковый и обратно — отдельная, более рискованная фича (означает создание/удаление CPT с потерей контента), сознательно не в этом эпике.
- **D18.2 — «Предметы» = подстраница `SubjectsMenuBuilder`/`SubjectPageCallbacks`.** Речь именно про этот пункт верхнего меню (список заданий/статей предмета); группы/курсы/зачисление находятся в других разделах меню и не затрагиваются.

### Оценка сложности (по итогам разведки кода)

- Точка ветвления одна: `SubjectController::registerForSubject()` — сейчас безусловно регистрирует 6 CPT + фиксированную таксономию для каждого предмета из `fs_lms_subjects_list`. Добавить условие «если `tasks_count === 0` — не регистрировать `tasks`/`articles`/`task_number`» технически несложно.
- Проблема — потребители: ~20 мест кода (менеджеры, коллбеки, сервисы: `SubjectDataCallbacks`, `SubjectPageCallbacks`, `TaskManager`, `TaskContentCallbacks`, `TemplateCallbacks`, `TemplateManagerCallbacks`, `ArticleService`, `TaskTypeService`, `LessonAuthoringService`, `WorkAuthoringService`, `SubjectDeletionService`, `SubjectExportService`, `TaxonomySettingsCallbacks`, `Enqueue` и др.) молча полагаются на то, что `PostTypeResolver::tasks($key)`/`articles($key)` существуют для любого известного ключа предмета. Большинство идёт через `WP_Query`/`get_posts()` — на несуществующий post_type просто вернёт пусто (не упадёт), но:
  - `LearningMenuController::subjectBankSlug()` жёстко ссылается на CPT первого доступного предмета учителя — нужен guard;
  - нативный edit-экран несуществующего CPT отдаёт WP-ошибку «Invalid post type» — нужно физически убрать пункты меню, а не просто спрятать их визуально;
  - `SubjectPageCallbacks::subjectPage` безусловно строит список-таблицы для `tasks`/`articles` CPT предмета — нужно скрывать саму подстраницу, а не рендерить пустую таблицу.
- Boilerplates/шаблоны (`BoilerplateRepository`) уже безопасны — все выборки по умолчанию возвращают `array()`/`null`, если для предмета ничего не задано.
- Межпредметных ссылок на задания сейчас нет вообще: `LessonAuthoringService`/`WorkAuthoringService` берут шаги либо из CPT **того же** предмета, либо из глобального банка `fs_lms_problems` — безбанковый предмет уже сегодня может ссылаться на общий банк без каких-либо изменений.

**Вывод:** умеренно инвазивно, но контролируемо — одна точка регистрации, но десятки точек молчаливого предположения «CPT есть». Отделить «безболезненно прямо сейчас» нельзя — нужен явный список guard'ов, а не одна ветка кода.
