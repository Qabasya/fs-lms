# Модульная архитектура FS LMS (in-plugin модули + флаги)

> Целевая архитектура и план перевода монолита в набор изолируемых модулей,
> загружаемых частями. **Стратегия зафиксирована: in-plugin модули + фича-флаги**
> (не отдельные WP-плагины). Эталон изолированного модуля — `Inc\Modules\AdSync`
> (см. `WpToADTasks.md`). Этот документ — карта при распиле и свод правил для нового кода.

---

## 0. TL;DR

- Плагин = **стабильное Ядро (Kernel)** + набор **модулей**, каждый со своим bootstrap, флагом, таблицами, конфигом.
- Два класса модулей: **слоистое ядро** (Content → Enrollment → Lms, зависят вниз) и **опциональные листья**
  (SocialAuth, Notifier, AdSync, будущие Личные кабинеты — нулевые входящие зависимости, связь только через хуки).
- Любой модуль **отключается одним флагом** (3 уровня) и **вырезается** удалением каталога + строки.
- **Главное правило на будущее:** новый функционал пишем так, чтобы его можно было **полностью отключить
  без потери предыдущего** (см. §4). Ядро никогда не ссылается на модуль.

---

## 1. Два класса модулей

**A. Слоистое ядро** — реальные зависимости кода идут вниз, отгружается «до уровня N»:
```
Kernel ── Content ── Enrollment ── Lms
```
Снять нижний слой без верхнего нельзя (Lms реально использует `GroupsRepository`/`StudentRecordRepository`
из Enrollment и `LessonManager` из Content — это корректная нисходящая зависимость).

**B. Опциональные листья** — в них никто не входит, связь с ядром только через хуки/публичные сервисы,
включаются/вырезаются свободно:
```
SocialAuth · Notifier(Telegram) · AdSync · [будущее: Cabinet — личные кабинеты]
```

Разница принципиальная: **листья изолируются дёшево**, **границы внутри ядра — дорого** (распутать ссылки).

## 2. Карта модулей и зависимостей

### 2.1. Граф
```
                 ┌───────── SocialAuth (лист)
Kernel ◄── Content ◄── Enrollment ◄── Lms
   ▲           ▲            ▲           ▲
   │           │            │           └── Cabinet (лист, будущее) ── личные кабинеты
   │           │            └── AdSync (лист) ── события Enrollment → провижн в AD
   └── Notifier (лист) ── шина событий → Telegram
```
(стрелка = «зависит от». Листья зависят от Kernel и **подписываются на события**, а не дёргают классы.)

### 2.2. Таблица модулей
| Модуль | Внутри | Зависит от | Отгружается как |
|---|---|---|---|
| **Kernel** | Container, Init/ModuleManager, ServiceInterface, base, общие enums/traits, PiiCrypto, PluginLogger, миграции-раннер, шина событий, конфиг | — | всегда |
| **Content** | предметы, задания, статьи, работы, уроки, курсы (CPT + банки + конструктор) | Kernel | «только предметы/задачи/статьи» |
| **Enrollment** | заявки, persons, группы, PII, зачисление/отчисление | Kernel (+soft: `subject_key`→Content) | «+ регистрация» |
| **Lms** | программа группы, сдачи, контрольные, плеер, прогресс, кокпит | Kernel, Content, **Enrollment** | «+ полноценная ЛМС» |
| **SocialAuth** | OAuth-провайдеры, привязка аккаунтов | Kernel | лист |
| **Notifier** | Telegram-уведомления (подписчик шины) | Kernel + шина | лист |
| **AdSync** | провижн учёток в AD через заявки | Kernel + события Enrollment | лист |

### 2.3. Будущие модули (пишем сразу как листья)
- **Cabinet (личные кабинеты)** — фронтовые страницы/шорткоды личного кабинета ученика/родителя/препода.
  **Должен быть листом:** читает данные через публичные API/хуки Enrollment+Lms; его **отключение не ломает**
  заявки/ЛМС (они работают и без кабинетов — как сейчас). См. правило §4.1.
- Любая новая фича — сначала вопрос: «это лист или слой ядра?» По умолчанию — **лист**.

> Content при желании под-разбивается: `ContentBank` (предметы/задания/статьи) + `Authoring`
> (работы/уроки/курсы) — для самого маленького релиза.

## 3. Механика

### 3.1. `ModuleInterface` + `ModuleManager` (вместо плоского `getServices()`)
Сейчас `Init::getServices()` — **плоский список из 55 сервисов**. Заменяем на модули:
```php
interface ModuleInterface {
    public function id(): string;          // 'lms'
    public function dependsOn(): array;    // ['content','enrollment']
    public function isEnabled(): bool;     // фича-флаг (config + wp-config константа)
    public function services(): array;     // ::class сервисов модуля (то, что сейчас в getServices)
}
```
`ModuleManager`: собрать модули → отфильтровать **включённые с удовлетворёнными зависимостями** →
топологическая сортировка → передать их `services()` в `Container`. 55 строк просто переезжают
в `services()` своих модулей. Включён Lms, выключен Enrollment → admin-notice
«Lms требует Enrollment», без молчаливой полузагрузки.

### 3.2. Фича-флаги — «одна кнопка» (3 уровня; как у AdSync)
1. **Runtime:** `fs_lms_plugin_config['<module>_enabled']` (тумблер в админке) → `isEnabled()=false` →
   модуль не регистрирует ничего (ноль хуков/cron/таблиц).
2. **Deploy:** константа в `wp-config.php` (напр. `FS_LMS_MODULE_LMS=false`) — перекрывает тумблер.
3. **Release:** удалить каталог `inc/Modules/<Name>/` + одну строку регистрации модуля.

### 3.3. Правила связности (это и есть изоляция)
- **Ядро → модуль: НИ ОДНОЙ ссылки.** Kernel/нижние слои не знают про верхние и про листья.
- **Листья:** запрещён `use Inc\Modules\Other\Concrete`; только **WP-хуки** (`do_action`/`apply_filters`) +
  публичные сервисы/контракты Kernel.
- **Слои ядра:** инъекция сервисов **вниз** разрешена (Lms берёт `GroupsRepository`); **вверх — никогда**
  (для обратного направления — `do_action`).
- **Точки соприкосновения слоёв — через интерфейсы в `Kernel/Contracts`** (напр. `GroupReadModel`,
  реализует Enrollment, потребляет Lms) — граница явная и подменяемая.

### 3.4. Владение данными и миграциями
- Сейчас одна монолитная `Migration_1_0_0`. Цель: **каждый модуль владеет своими таблицами** и отдаёт
  фрагмент миграции; раннер агрегирует фрагменты **включённых** модулей.
- Модуль **не пишет** в таблицы/опции/мету другого модуля напрямую — только через его публичный репозиторий/сервис.
- Общие enums (`OptionName`, `TableName`, `Capability`, `Nonce`, `AjaxHook`): дробим — фиче-специфичные
  значения едут в свой модуль, в Kernel остаётся только реально общее.

### 3.5. Раскладка каталогов
```
inc/
├── Kernel/    (Container, Init, ModuleManager, ServiceInterface, base, общие enums/traits,
│               PiiCrypto, PluginLogger, migrations-runner, event bus, config)
└── Modules/
    ├── Content/      Enrollment/      Lms/        ← слоистое ядро
    └── SocialAuth/   Notifier/   AdSync/  Cabinet/ ← листья
```
Каждый модуль: свои `Controllers/ Services/ Repositories/ DTO/ Enums/ Migrations/ Config/` + `XxxModule.php` (bootstrap).

## 4. Правила написания кода на будущее (обязательно)

> Пишем **с оглядкой на модульное разделение**, даже пока код ещё в монолите. Дёшево соблюдать
> сейчас — дорого распутывать потом.

### 4.1. Новый функционал должен полностью отключаться без потери предыдущего
- Каждая новая фича — **отдельный модуль с флагом** и изолированным bootstrap; при выключении `register()`
  сразу `return` → нулевой след.
- **Пример (личные кабинеты):** Cabinet — лист. Заявки, зачисление, ЛМС обязаны работать при выключенном
  Cabinet **ровно как сейчас**. Кабинет только добавляет фронтовые страницы поверх существующих данных,
  ничего в ядре не «переключает на себя».
- Проверка: для каждого модуля — тест/чек «выключили → остальное работает».

### 4.2. Однонаправленная зависимость
Зависимость всегда **вниз/в Kernel**, никогда наверх. Ядро не ссылается на фичи. Лист не ссылается на лист.

### 4.3. Связь между фичами — через швы, не через классы
- Между листом и ядром / между листьями — **только** `do_action`/`apply_filters` или интерфейсы Kernel.
- **Не** плодить `if ( class_exists( OtherModule\X ) )`-спагетти. Нужна интеграция — заведи хук/контракт.
- Хук-швы в ядре делаем **обобщёнными** (`fs_lms_application_created`, `fs_lms_student_enrolled`…),
  не под конкретный модуль — без модуля они no-op.

### 4.4. Владей своими данными
Фича заводит **свои** таблицы/опции/мету (+ фрагмент миграции) и не лезет в чужие напрямую.

### 4.5. Graceful absence
Если фича A **улучшает** B — B обязана работать без A (через `has_action`/хуки/наличие сервиса, не через
жёсткую зависимость).

### 4.6. Фиче-специфичное — в фиче
Enums, nonce, AJAX-хуки, capabilities новой фичи живут в её модуле; в Kernel — только общее.

### 4.7. Регистрация хуков — в контроллере модуля
Как и сейчас (CLAUDE.md): `add_action`/`add_filter` только в контроллерах, теперь — контроллерах **модуля**.

## 5. Что разделяемо «дёшево» vs «дорого» (честно)

| | Сложность | Почему |
|---|---|---|
| Листья (SocialAuth, Notifier, AdSync, Cabinet) | **дёшево** | аддитивны, нулевые входящие зависимости, связь хуками |
| Граница Content / Enrollment / Lms | **дорого** | подтверждённые перекрёстные ссылки (7 файлов Lms → Enrollment-репозитории), общие enums |
| Общие данные | **самое дорогое** | persons/groups/applications/group_lessons + subjects(option) + монолитная `Migration_1_0_0`; нужно назначить владельцев таблиц и разрезать миграцию |

Вывод: начинаем с листьев (быстрая отдача), границы ядра трогаем инкрементально.

## 6. План миграции монолита (инкрементально, без «переписать всё»)

1. **Каркас:** ввести `ModuleInterface` + `ModuleManager`; **сгруппировать существующие 55 сервисов** по
   модулям-bootstrap — без смены поведения, только структура. Низкий риск, сразу даёт карту.
2. **Листья:** вынести SocialAuth, Notifier, AdSync в `inc/Modules/...` (AdSync — уже по плану `WpToADTasks.md`).
3. **Флаги:** включить per-module флаги (3 уровня §3.2) + admin-notice о неудовлетворённых зависимостях.
4. **Границы ядра:** провести Content/Enrollment/Lms; перенести файлы; сузить межслойные точки до
   интерфейсов `Kernel/Contracts`.
5. **Миграции:** разрезать монолитную `Migration_1_0_0` на фрагменты модулей; раннер агрегирует включённые.
6. **Enums:** раздробить общие enums по владельцам.
7. **(Позже, опц.)** вынести модуль в отдельный плагин, если понадобится раздельная поставка — границы
   делают это дёшево.

## 7. Решение: in-plugin модули + флаги (зафиксировано)

- **Выбрано:** одна кодовая база, чистые внутренние границы, отключение конфигом, вырезание каталогом.
  Минимум накладных расходов.
- **Отвергнуто (пока):** отдельные WP-плагины — оверхед (заголовки, версии, проверка «ядро активно»,
  кросс-плагинная автозагрузка). Вернёмся, только если нужны **отдельные дистрибутивы**.
- **Гибрид-задел:** границы модулей строим так, чтобы лист можно было позже вынести в отдельный плагин
  без переписывания (зависимость только на публичный API Kernel + хуки).
