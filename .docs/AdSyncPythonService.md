# AD Sync — техническое задание для Python-сервиса

> Для разработчика, который пишет/поддерживает **Python-поллер**, создающий учётные записи в Active
> Directory по заявкам с сайта. WP-сторона (модуль `Inc\Modules\AdSync`) полностью готова и не меняется
> под этот сервис — данный документ описывает требования **к Python-сервису**: что он получает от WP,
> какие у него самого точки конфигурации, и что конкретно он должен уметь делать в AD.
>
> Полный REST-контракт (auth, точные форматы запросов/ответов, коды ошибок, рабочий пример
> клиента-поллера) — `.docs/FS_LMS_API.md`, раздел «AdSync». Этот документ его не дублирует, а
> описывает бизнес-логику и требования вокруг него.

---

## 1. Роль сервиса и модель взаимодействия

- **Домен-контроллер в локальной сети без белого IP.** WP — публичный сайт, который никогда не
  достаёт до Python (push невозможен). Поэтому модель — **pull**: Python сам, исходящими HTTPS-запросами,
  опрашивает WP, забирает задания и отчитывается о выполнении.
- Сервис — это **клиент-поллер** (демон/cron/systemd-timer), не веб-сервер и не принимает входящих
  запросов. Рекомендуемый стек: `requests`/`httpx` + `ldap3` (LDAP/LDAPS) + планировщик
  (`APScheduler` или системный cron).
- Три цикла опроса:
  1. **Задания** (часто, интервал ~2–5 с — от момента заявки до входа ученика должны пройти секунды):
     `GET /ad/jobs` → выполнить в AD → `POST /ad/ack`.
  2. **Сверка-«пылесос»** (редко, раз в час/сутки): `GET /ad/active-usernames` → отключить в
     управляемой OU всех, кого нет в списке.
  3. Нет третьего цикла — WP ничего не пушит и не ждёт от Python синхронного ответа.

## 2. Аутентификация

HMAC на каждом запросе, заголовки `X-Fs-Timestamp` + `X-Fs-Signature`, секрет — общий с
`FS_LMS_AD_HMAC_SECRET` из `wp-config.php` сайта (одно и то же значение в `.env` Python-сервиса).
Точная формула и пример подписи — `.docs/FS_LMS_API.md` §2. Секрет генерируется кнопкой в админке
секции «Синхронизация с доменом (AD)» — единственный источник истины по текущему значению.

## 3. Что сервис получает от WP

### 3.1. `GET /ad/jobs?limit=50`

Возвращает массив готовых заданий, разный набор полей в зависимости от `event`:

| `event` | Поля payload | Что делать |
|---|---|---|
| `provision` | `id, event, idempotency_key, username, password, first, last, subject_key` | создать учётку, задать пароль, добавить в security-группу направления |
| `promote` | `id, event, idempotency_key, username` | перенести учётку в OU/группы зачисленного |
| `deprovision` | `id, event, idempotency_key, username` | отключить учётку |

Важно:
- **Пароль приходит только в `provision`** и только в момент выдачи задания — WP не хранит его в
  очереди, а расшифровывает из заявки на лету. Для `promote`/`deprovision` пароля нет и не будет.
- **`subject_key`** — стабильный слаг направления (напр. `inf`, `math`), **не** DN группы. Карту
  `subject_key → DN security-группы` WP не хранит — её держит **сам Python** (конфиг/env). Если
  придёт незнакомый `subject_key`, это повод залогировать предупреждение и создать учётку без группы
  направления, а не упасть.
- Задание может **исчезнуть** из выдачи без объяснений (например, заявку успели удалить до выдачи
  задания) — это не ошибка, просто WP пропускает такую строку, ack от Python не требуется и не ожидается.
- `limit` от 1 до 200 (по умолчанию 50) — сервис должен уметь принять до 200 заданий за раз, если
  очередь накопилась (например, после простоя сервиса).

### 3.2. `POST /ad/ack`

Тело `{ "id": <int>, "status": "done"|"failed", "error"?: "...", "sam_account_name"?: "..." }`.

- `status` считается успехом при значениях `done`, `ok`, `sent`, `success` — любое другое (включая
  пустое) трактуется как неудача. Рекомендуется всегда слать именно `done`/`failed`.
- **`sam_account_name` WP на сегодня не использует** (поле принимается эндпоинтом, но нигде не
  сохраняется) — присылать его не обязательно, это задел на будущее, а не рабочий канал передачи данных.
- **Ack обязателен для каждого обработанного задания** — и в случае успеха, и в случае ошибки.
  Если Python не пришлёт ack вовсе (упал посреди обработки), задание останется `pending` и будет
  выдано повторно на следующем поллинге — обработчик должен переносить повторную выдачу спокойно
  (см. §5 «Идемпотентность»).

### 3.3. `GET /ad/active-usernames`

Возвращает `{ "usernames": ["i.petrov", …] }` — авторитетный список логинов, которые **обязаны**
оставаться активными прямо сейчас: активно зачисленные ученики + заявки в живых статусах (ожидание
подтверждения родителем / на проверке / в процессе зачисления). Истёкшие, отправленные в корзину и
уже деправижненные заявки в список не входят.

Сервис должен на управляемой OU **отключить** (не обязательно удалить) все учётки, которых нет в
этом списке — это закрывает случаи, когда событие `deprovision` потерялось или человек был удалён
из БД напрямую в обход событий.

## 4. Что сервис должен уметь делать

### 4.1. Операции в AD (ldap3)

- Подключение под **сервис-аккаунтом наименьших прав**, с делегированием только на управляемую OU.
  Пароль сервис-аккаунта — в секретах/env, не в коде.
- **Создание (`provision`):** `objectClass=user`, `sAMAccountName=username`,
  `userPrincipalName=username@<домен>`, `cn`/`displayName` из `first`+`last`; создаётся сразу
  **включённым** (`userAccountControl=512`), в `OU=Pending`.
- **Пароль:** атрибут `unicodePwd` (значение в кавычках, UTF-16LE) — **обязательно по LDAPS (порт 636)**,
  открытый LDAP для установки пароля AD не примет.
- **Группа направления:** `modify` атрибута `member` группы (`MODIFY_ADD`) — DN пользователя
  добавляется в группу, соответствующую `subject_key` по конфигу Python. `memberOf` не пишется
  напрямую — это вычисляемый атрибут.
- **`promote`:** перенос объекта из `OU=Pending` в `OU=Active` (`modify_dn`) + при необходимости
  добавление в дополнительные группы зачисленного.
- **`deprovision`:** отключить (`userAccountControl=514`) и перенести в `OU=Disabled`
  (`modify_dn`) — **не hard-delete по умолчанию** (сохраняет аудит и возможность ручного разбора).
- **Сверка (`active-usernames`):** пройтись по управляемой OU, для каждой активной учётки проверить
  вхождение `sAMAccountName` в присланный список; если нет — отключить (как в `deprovision`).

### 4.2. Идемпотентность и повторная обработка

- `idempotency_key` в задании (`app:{id}` для provision, `deprovision:app:{id}`,
  `promote:person:{id}`) — используйте его как ключ локальной дедупликации/лога, а не полагайтесь
  только на `id` задания.
- Задание может прийти повторно, если предыдущий ack не дошёл до WP (сетевой сбой) или обработка
  упала на середине. Все операции в AD должны быть **безопасны при повторе**: `create` — проверять
  существование учётки перед созданием (или ловить ошибку «уже существует» и трактовать как успех/
  апдейт, не как фатал); `add member`/`modify_dn`/disable — идемпотентны по своей природе в LDAP
  (повторное добавление в группу или отключение уже отключённой учётки не должно быть ошибкой).
- **Ретраи и backoff — на стороне WP, без участия Python.** При `ack(failed)` WP сам увеличивает
  счётчик попыток и переотдаёт задание с экспоненциальной задержкой (1м → 2м → 4м → … до 1ч), после
  6 неудачных попыток задание помечается «мёртвым» и больше не появляется в `/ad/jobs` (нужно
  логировать `dead` у себя, дальше человеку разбираться вручную — WP уведомление о «мёртвых»
  заданиях наружу не шлёт).

### 4.3. Конфигурация, которую сервис должен завести сам

WP не хранит и не присылает ничего из этого — исключительно локальный конфиг/env Python-сервиса:

| Параметр | Назначение |
|---|---|
| `BASE_URL` | адрес сайта WP (`https://.../wp-json/fs-lms/v1`) |
| `FS_LMS_AD_HMAC_SECRET` | секрет подписи запросов (то же значение, что в `wp-config.php`) |
| DN'ы security-групп по `subject_key` | карта «направление → группа», WP её не хранит |
| DN'ы OU `Pending` / `Active` / `Disabled` | структура жизненного цикла учётки |
| Сервис-аккаунт AD | логин/пароль/DN для bind по LDAPS |
| Интервалы поллинга | задания (секунды), сверка (часы) |

### 4.4. Логирование и безопасность

- Структурный лог на каждое задание: `event`, `idempotency_key`, результат — **без пароля** (пароль
  нигде не логируется, даже в debug-режиме).
- Метрики: создано / отключено / ошибок — для мониторинга простоя сервиса (если поллер долго не
  забирает задания, очередь на WP-стороне просто растёт, сайт от этого не страдает, но ученики не
  получат доступ вовремя).
- HTTPS обязателен (эндпоинт `/ad/jobs` отдаёт пароль в открытом виде тела ответа под TLS).
  IP-allowlist на стороне WP не применяется (у локалки нет статического адреса) — единственная
  защита канала это HMAC + HTTPS.

## 5. Требования к самому домену (нужно от AD-администратора, не от кода)

- **Fine-Grained Password Policy** на OU учеников с ослабленной сложностью/минимальной длиной —
  иначе AD отклонит простой пароль, заданный учеником в форме, и `provision` будет постоянно падать
  с `ack(failed)`. WP пароль никак не валидирует и не навязывает политику.
- OU-структура `Pending` / `Active` / `Disabled` и security-группы по направлениям должны
  существовать заранее — их DN идут в конфиг Python (см. §4.3).
- Сервис-аккаунт с правами только на управляемую OU (создание/изменение/перемещение объектов,
  запись `unicodePwd`, изменение `member` нужных групп) — по принципу наименьших привилегий.
- Настройка прав на общие ресурсы, ярлыки, раскладку меню и т. п. через GPO/ACL по группам
  направлений — это разовая настройка домена, к рантайму сервиса не относится (сервис только
  добавляет ученика в нужную группу, всё остальное отрабатывают уже существующие политики AD).

## 6. Открытые вопросы (уточнить перед вводом в прод)

- Точные DN OU жизненного цикла и security-групп по направлениям.
- Формат `userPrincipalName` (домен после `@`) и наличие кастомного атрибута для хранения
  `person_id`/`application_id` (полезно для отладки соответствия «учётка AD ↔ заявка», не обязательно).
- Что делать с заданием, если `subject_key` не входит ни в одну известную группу (создавать без
  группы направления с предупреждением в лог — предлагаемое поведение по умолчанию).
