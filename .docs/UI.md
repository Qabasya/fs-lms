# UI.md — Унификация дизайн-системы админки

Цель: свести разнородные, но почти одинаковые UI-компоненты админки к одной системе
«родительский компонент → детали через параметры/модификаторы», максимально на токенах,
с адаптивной вёрсткой. Область — **только админка** (`src/scss/admin/`, `templates/admin/`,
`src/js/admin/`). Фронт (`/lms/*`) не трогаем.

---

## 1. Принципы

1. **Один родитель — варианты через параметры.** PHP-рендерер принимает `$args`/слоты; SCSS —
   один BEM-блок + модификаторы (`.fs-card`, `.fs-card--module`, `.fs-card--compact`). Новый вид =
   модификатор, а не новый компонент.
2. **Всё через токены `_variables.scss`** — цвета, отступы, радиусы, ширины, тени, типографика,
   transition. В компонентных SCSS — никаких сырых значений.
3. **НЕ плодить токены на мелкие отличия.** Если значение отличается от существующего токена на
   несколько пикселей (длины) или на пару оттенков (цвета) — **снапить к существующему токену**, а не
   заводить новый. Аудит — это в т.ч. *рационализация шкалы*: схлопнуть близкие значения к канону,
   а не механически заменить литералы на переменные.
4. **Адаптив по умолчанию.** Flexbox/grid + `flex-wrap` + `gap` + `clamp()`; для реакции на ширину
   **контейнера** (а не вьюпорта) — CSS container queries (`container-type: inline-size` + `@container`).
5. **Конструкторы — только обвес.** Большие билдеры используют общие токены + примитивы (поля,
   кнопки, рамки-карточки), но свою внутреннюю раскладку оставляют bespoke.
6. **JS-разметка — паритет с PHP.** Модалки и билдеры рисуют HTML строками в JS; для них нужны
   JS-двойники рендереров или единое соглашение по классам, иначе разъедется с PHP-версией.

## 3. Целевая библиотека примитивов

Покрывает выделенные UI-элементы. Каждый — один PHP-рендерер (`render_fs_*`) + один BEM-блок.

| # из списка | Примитив | Рендерер | Ключевые варианты/параметры |
|---|---|---|---|
| 1 Карточки | Card | `render_fs_card()` | `--module`, `--compact`; слоты header/body/footer |
| 2 Таблицы · 7 аккордеон | Table | `render_fs_table()` | `--accordion` (строки-аккордеоны), колонки-конфиг |
| 3 Плашка «ещё нет» | Empty state | `render_fs_empty()` | текст + опц. иконка/CTA |
| 4 Контейнер полей (ширина настраивается) | Field / Form | `render_fs_field()`, `render_fs_form()` | тип поля; ширина — модификатор/`--fs-field-width` |
| 5 Кнопка «+» | Add button | `render_fs_add_button()` | вариант кнопки (icon-only) |
| 6 Badge, toggle | — | ✅ есть | — |
| 8 Строка фильтрации | Filter bar | `render_fs_filter_bar()` | слоты под поля/селекты, выравнивание |
| 9 Описание · 10 кнопки справа | Page header | `render_fs_page_header()` | `title` + `description` + `actions` (справа) |

«Ширина настраивается» (п.4): модификаторы `--narrow/--wide/--full` (значения — токены ширины)
или CSS-переменная `--fs-field-width` на обёртке.

### Большие компоненты
- **Конструкторы** (`_course-builder.scss` + ~20 `.fs-cb-*`, `_assessment-builder.scss`,
  `_step-editor.scss`, `_slot-builder.scss`) и **конструктор урока** — chrome-only: принимают токены +

## 5. Правила токенизации (важно)

**Токенизировать** (канон в `_variables.scss`): палитра (text/bg/border/accent/muted/danger/success),
шкала отступов (напр. 4/8/12/16/24/32), радиусы (sm/md/lg/pill), тени (card/modal), типографика
(размеры/веса/line-height/семейства), transition, ширины контейнеров.

**НЕ заводить новый токен**, если значение:
- отличается от существующего на несколько пикселей (длины/отступы/радиусы) → снапить к ближайшему;
- отличается на пару оттенков от существующего цвета → использовать существующий;
- встречается 1–2 раза как осознанная единичная деталь → допустим локальный литерал с комментарием,
  но сначала проверить, не снапится ли к канону.

Итог аудита — *меньше* значений, чем сейчас, и почти все из них именованные.

## 6. Методология поиска неиспользуемых классов

1. Извлечь определения: `grep -rhoE '\.fs-[a-z0-9_-]+' src/scss/admin` → уникальные селекторы.
2. Для каждого проверить вхождение (без точки) в `templates/admin/` и `src/js/admin/`.
3. Отсутствует в обоих → **кандидат**, не автоудаление: вручную исключить динамические случаи
   (классы, собираемые в JS из кусков, `data-`-управляемые стили, состояния `is-/has-`).
4. Симметрично: классы в разметке, под которые нет ни SCSS-правил, ни JS-обработчиков → удалить из разметки.

Definition of done аудита: ноль «висячих» классов в обе стороны (SCSS↔разметка), список удалённого — в PR.

## 7. Риски и ограничения

- Сквозной рефактор: правка общего примитива меняет вид везде → постраничная визуальная проверка обязательна.
- Только инкрементально; big-bang исключён.
- WP-admin: своя сетка и схлопывающийся сайдбар меняют доступную ширину — учитывать при container-queries.
- Конструкторы имеют минимальные ширины: трёхколоночную раскладку в одну без логики не сложить — там
  адаптив ограничен переносом второстепенных панелей.
- JS-разметка (Фаза 4) — самый недооценённый объём; без неё модалки/билдеры разойдутся с PHP-примитивами.

## 8. Definition of done (вся работа)

- Каждый UI-элемент из списка — один примитив (рендерер + BEM-блок), варианты через модификаторы.
- В компонентных SCSS нет сырых значений (кроме осознанных единичных литералов с комментарием).
- Нет неиспользуемых классов (SCSS↔разметка) и мёртвых правил.
- Примитивы адаптивны (flex/grid/wrap; container-queries где нужно).
- Конструкторы используют токены + примитивы для обвеса.
- `npx gulp build` чист, страницы визуально консистентны.
